## P1 · 本周优先（真实价值/安全影响，范围有界）

### P1-1 免费课「拿走↔移除」循环刷作者创作激励（积分铸造）✅已亲验

- **位置**：`src/lib/credit-trade.ts:83,95-112`（`collectFreeCourse` 靠 `CoursePurchase.create` 撞唯一约束做幂等，成功即给作者 `authorBonus`）× `src/app/api/market/collect/route.ts:187-190`（DELETE 对免费课**真删** `CoursePurchase`）
- **根因**：激励幂等键是「CoursePurchase 永久存在」，但免费课的 uncollect 会删掉它。删掉后再 collect → `create` 重新成功 → 作者**再次 +2**。两个设计相互冲突。
- **可利用场景**：A（作者，发一门免费课）+ B（收集者）。B 循环 `POST`(拿走)→`DELETE`(移除)→再拿走，每轮给 A 凭空 +2。限流键独立（collect/uncollect 各 60/h）→ **≈120 积分/h，无生命周期上限**。积分可真钱购买（¥6=60 分）且能驱动真实 DeepSeek 造课 → 真实价值泄漏。
- **修复**：二选一——① 免费课创作激励改为「每 (collector,course) 一次的持久幂等」：新增 `FreeCollectBonus @@unique([collectorId,courseId])` 作激励去重键，uncollect 删 CoursePurchase 时**不删**该记录；② 更省事：免费课 uncollect 时不删 CoursePurchase（仅移出书架/隐藏，与付费课一致）。推荐 ②。
- **工作量**：② 约 20 分钟；① 约 1 小时（含迁移）。回归：`tests/` 补一条「拿走→移除→再拿走，作者仅得一次」。

### P1-2 `GET /api/progress/me` 下发完整课件正文 + 潜在付费门泄漏 ✅已亲验

- **位置**：`src/app/api/progress/route.ts:75-80`（`include:{ course:true, lesson:true }`，无 `select`、无分页、无权益门）
- **根因**：原样返回整行 `lesson`，含 `blocksJson`/`htmlJson`/`articleMd`/`videoScriptJson` 等**完整正文**大文本；而正常读课走的 `getLessonForUser` 会按 access 把正文置空。此端点被 iOS 消费。
- **后果**：① 带宽/内存 P1——重度用户单次响应巨大；② **付费门泄漏**：任何在某章留有 `LearningProgress` 的用户（含退订后、含免费预览/被拿走时写入首章进度）都能从进度列表拿到该章完整正文，绕过 `getLessonForUser` 的按权益置空。
- **修复**：改 `select` 只取列表展示字段（课程标题/slug、章标题、progressPct、lastPlayedAt），**移除所有正文字段**；加分页（cursor/take）。正文一律回 `getLessonForUser` 走权益门。
- **工作量**：30 分钟 + 契约冒烟确认 iOS 进度页字段未破。

### P1-3 积分预检低估真实成本 → 订阅用户单次越额免单 ✅逻辑已核

- **位置**：`src/lib/credits.ts:99-101`(`estimateCredits` 写死 `approxTokens=3000`、不传 model → `costWeight=1`)、`:160-167`(`assertCanSpend`)、`src/app/api/ai/generate-course/route.ts:53-55`、`src/lib/course-gen.ts:845-861`(`runCourseGenBackground` 逐节扣费但不再预检)
- **根因**：① 门槛按固定 3000-token×权重1 估，**不含所选模型权重（Opus 权重达 3）与精修档**；真实逐节 `maxTokens` 8000、bespoke HTML 16000。一次 premium HTML 实扣 ≈81 分 vs 门槛 5 分（≈16×）。② 造课**只在入口预检一次 3 分**，随后对全部 8 节 + 最多 6 节 HTML 逐个扣费而不再校验余额。
- **可利用场景**：最低档订阅用户余额仅剩 3 分即可发起一门 premium 造课，把余额刷成 -N（可达上百），运营方 DeepSeek/NewAPI 实付这笔 token。`recordLlmSpend` 允许负余额、下次才拦，故每个「恢复到门槛以上」周期可越额一门。与 `route.ts:53-55` 注释「按最坏成本设门槛」矛盾。
- **修复**：① `estimateCredits`/`assertCanSpend` 传入 `model`+`qualityTier` 按真实 `costWeight` 与典型 `maxTokens` 计门槛，造课门槛再乘预计节数；② 更稳：`runCourseGenBackground` 每节 `generateLessonCore` 前 `assertCanSpend`，余额不足即中止后续节并置 `genStatus`，把「一次授权整课」改「逐节授权」。
- **工作量**：① 30 分钟；②+① 约 1.5 小时 + AI 回归（需 DeepSeek 余额）。

## P2 · 排期（不急但值得做）

### 支付 / 权益正确性

**P2-1 优惠券配额无下单预留、无每人上限 → 可超发/重复用**
- 位置：`src/lib/payment.ts:149-161`(applyCoupon 只读 `redeemedCount` 判满、不预留)、`:311-338`(webhook 才自增，且名额满时注释 `coupon_oversell` 仍放行)
- 根因/危害：折扣在**下单**时计入金额，`redeemedCount` 到**支付回调**才自增。① 并发超发：N 人在任何人支付前同时下单都读到未满、全部拿折扣；`maxRedeem` 沦为事后统计。② 无 per-user 约束，同一用户可对同券反复下单。属营销预算直接泄漏。
- 修复：`createCheckoutSession` 落 pending 单时即原子预留（`updateMany(redeemedCount<maxRedeem){increment:1}`，失败拒单），退款/超时取消时释放；若券语义「每人一张」加 `CouponRedemption @@unique([couponId,userId])`。

**P2-2 Apple 退款通知过度回收：抹掉同订阅上未退款的已付周期**
- 位置：`src/app/api/iap/notifications/route.ts:104-132`（`:122-131` 直接把 `currentPeriodEnd=now`+`refunded`）
- 根因：Web/IAP/管理续期都复用同一条 scope="all" 订阅行，但退款把整条清零，不像 `payment.ts:416-437` 的 `rollbackOnePeriod` 只精确回退一期。
- 危害（用户受损方向）：① Web 购一月 + IAP 续一期后，仅退 IAP 那笔 → Web 已付月份一并作废；② 纯 IAP 多次续费退某一期 → `candidateOrderIds` 连带把 original 也标 refunded、整条订阅归零。
- 修复：改用 `rollbackOnePeriod(该单周期, sub.currentPeriodEnd)`，仅当回退后 ≤now 才置 refunded；且只处理 `order.subscriptionId` 指向的那条 sub。

**P2-3 IAP 校验用 `in` 判 productId → 原型链键穿透**
- 位置：`src/app/api/iap/verify/route.ts:121,159`（`if (productId in CREDIT_PRODUCTS)` / `SUBSCRIPTION_PRODUCTS`）
- 根因：`in` 命中 `Object.prototype` 继承键。传 `productId="constructor"/"toString"` 判真：积分分支 `amount=函数` 致 500；订阅分支 `billingPeriod=undefined` 使 `plan.findFirst({where:{billingPeriod:undefined}})` 退化为不过滤、随便建一份订阅。
- 边界：生产被 Apple `validateClaims` 挡（productId 与签名载荷不符）；仅未配 Apple 的非生产 mock 直发路径可触发。属健壮性/纵深防御。
- 修复：改 `Object.hasOwn(CREDIT_PRODUCTS, productId)`（或 `hasOwnProperty.call` / 改 `Map`）。

### 数据层 / 性能（扩容期塌陷）

**P2-4 `busy_timeout` 仅作用于连接池的一条连接**
- 位置：`src/lib/db.ts:25`（`PRAGMA busy_timeout=5000` 经单次 `$queryRawUnsafe`，只对承接它的那条连接生效；WAL 是库持久属性无碍，但 busy_timeout 是每连接易失属性）

**P2-5 `CourseUpdateLog` 完全无索引（热点路径全表扫）**

**P2-6 一批无 `take`/无分页/无时间窗的全表查询**

### 认证 / 限流（纵深防御）

**P2-9 登录接口基于时序的用户枚举**
- 位置：`src/app/api/auth/login/route.ts:24`（`!user || ... || !verifyPassword(...)`：账号不存在时 `||` 短路跳过昂贵 scrypt，存在但密码错则跑完整 scrypt，时间差数十毫秒）
- 危害：文案已统一防枚举，但时序差仍可区分「邮箱/手机号是否注册」；限流只减缓不消除。
- 修复：账号缺失时对一个常量假哈希跑一次等价 `verifyPassword`，对齐两分支耗时。

**P2-10 会话 token 明文入库（与已哈希的重置 token 不对称）**
- 位置：`src/lib/session.ts:91-93,134-140`（token 原文写入 `Session.id` 并明文查）vs `password-reset/route.ts:36` 存 `sha256(token)`
- 危害：只读级 DB 泄露（注入取数/备份外泄）即可拿全部在线会话 token 直接重放至 30 天过期；重置 token 因已哈希不可逆用。
- 修复：存 `sha256(token)` 作查找键，原文只留 cookie/Bearer，与重置 token 对齐。

**P2-11 IP 限流无条件信任 `X-Real-IP`，盖过已加固的 XFF 逻辑**
- 位置：`src/lib/rate-limit.ts:52-65`（`pickClientIp` 先无条件返回客户端可传的 `x-real-ip`，其后才是「取倒数第 N 跳」的 XFF 加固）
- 危害：若反代未强制覆盖入站 `X-Real-IP`，攻击者每请求换值即得全新限流桶——正是 XFF 想堵的洞。影响仅 IP 维度的面：注册（薅注册积分）、找回密码申请（向任意邮箱洪泛）；登录另有账号维度限流兜底。
- 修复：仅在确有可信反代设置时才信任 `x-real-ip`（env 开关），否则统一走 XFF「倒数第 `TRUSTED_PROXY_HOPS` 跳」。
- 附注：限流为进程内内存（`rate-limit.ts:9`）、inflight 锁同（`ai/inflight.ts`），多实例/Serverless 跨实例失效——代码已注明「生产换 Redis」，属已知取舍。
