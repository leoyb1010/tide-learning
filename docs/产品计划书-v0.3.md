# 「潮汐学习」产品计划书 v0.3

> 用途：可直接交给产品 Agent / 设计 Agent / 前后端 Agent 拆解执行。  
> 基础：基于 v0.2 计划书增强。  
> 版本：v0.3 · 2026-07-02  
> 覆盖端：Web / iOS / Android 预留三端架构；MVP 优先 Web。  
> 核心判断：原方向成立，但需要补齐 MVP 边界、后台 CMS、数据模型、订阅状态机、埋点、内容生产 SOP、合规口径和验收标准。

---

## 0. v0.2 评估结论

### 0.1 已经做得好的部分

1. **定位清晰**：不是单课买断，而是“滚动更新的订阅制学习流”。
2. **差异点明确**：需求排行榜、滚动上新、学习笔记联动、全龄双模式，具备区别于传统知识付费平台的基础。
3. **设计方向正确**：高级感、呼吸感、动效、深浅色、长辈模式都有明确要求。
4. **技术方向基本合理**：Web SSR、权益中心、排行榜、AI 网关、三端 token 共享，这些方向可以保留。

### 0.2 主要缺口

| 缺口 | 问题 | v0.3 补强 |
|---|---|---|
| MVP 过宽 | 一开始想覆盖三端、三类人群、多类课程、AI、笔记、共创、家庭卡，执行容易失控 | 明确 P1 / P2 / P3，P1 只验证“订阅 + 学习 + 共创 + 基础笔记” |
| Agent 执行颗粒度不足 | 页面写得多，但缺少验收条件、数据结构、接口、状态、空态、异常态 | 新增 PRD 验收口径、数据表、事件埋点、后台 CMS、状态机 |
| 内容生产机制不够硬 | 说“滚动更新”，但缺少课程制作 SOP、更新 SLA、质量审核流程 | 新增内容工厂 SOP、冷启动课程包、更新节奏、质量红线 |
| 合规表达有风险 | “AI 问诊”容易被理解为医疗服务 | 统一改成“就医前信息整理 / 健康信息素养”，禁止诊断、处方、用药建议 |
| 订阅设计不够工程化 | 包月、连续包月、年度都有，但缺状态机、退款、宽限期、跨端恢复购买 | 新增 entitlement 状态机和支付渠道处理规则 |
| 长辈模式需要更具体 | 有适老方向，但缺功能降噪和诱导防护 | 新增长辈模式首页、支付、字体、语音输入、家人协助验收 |

### 0.3 最大建议

**P1 不要同时追求“全龄全品类全端”。**  
产品最终可以覆盖 18–65 岁，但 MVP 必须先用最容易付费、最容易复购、最容易更新的内容验证模式。

建议 P1 采用：

1. **主力内容**：AI 实用技能 + 雅思/英语备考。
2. **生活线试点**：只上一门“长辈 AI 入门 / 就医前信息整理 / 防诈骗”代表课，不做大规模铺开。
3. **端优先级**：Web 优先，移动端先做响应式 H5；iOS / Android 放 P2。
4. **AI 功能**：P1 不做复杂 AI 助教，只预留入口；P2 上线 AI 笔记整理和课程内问答。

## 1. 产品定位升级

### 1.1 一句话定位

**潮汐学习是一个全龄订阅制学习平台：用户按月订阅，解锁持续更新的体系化课程，并通过需求投票决定平台下一批课程。学习、笔记、复习和上新提醒形成长期留存闭环。**

### 1.2 核心差异

| 维度 | 传统知识付费 | 免费视频平台 | 潮汐学习 |
|---|---|---|---|
| 付费方式 | 单课买断 | 免费 + 广告 | 月费 / 年费订阅 |
| 内容形态 | 上架后基本固定 | 碎片化 | 课程持续更新，有更新日志 |
| 课程决策 | 平台/讲师主导 | 流量算法主导 | 用户提交 + 投票 + 排行榜驱动 |
| 留存资产 | 看完即走 | 收藏难复用 | 笔记、锚点、复习、学习路径 |
| 用户年龄 | 多偏年轻职场人 | 年轻人居多 | 18–65 岁，标准模式 + 长辈模式 |
| 家庭场景 | 弱 | 弱 | 家庭订阅，子女可为父母开通 |

### 1.3 产品护城河

1. **内容更新机制**：每门课程必须有“更新时间、更新日志、下一次预计更新”。
2. **用户共创机制**：用户投票决定课程选题，提高参与感和续订理由。
3. **笔记资产机制**：笔记永久保留，停订后也可查看和导出。
4. **全龄体验机制**：标准模式服务年轻人，长辈模式服务 45–65 岁用户及家庭代付。
5. **内容可信机制**：课程必须显示讲师、审核人、版本号、更新时间，健康/财务类内容必须额外审核。

## 2. MVP 边界

### 2.1 P1 目标

**在 90 天内验证：用户是否愿意为“持续更新的课程库 + 需求共创 + 基础笔记”付订阅费。**

### 2.2 P1 必做

| 模块 | P1 范围 |
|---|---|
| Web 首页 | 完整营销首页、课程展示、订阅转化、登录入口 |
| 用户系统 | 手机号 / 邮箱登录，微信登录可选 |
| 课程库 | 课程列表、分类、搜索、课程详情、章节大纲 |
| 播放/学习 | 视频/图文课、进度记录、试学、付费墙 |
| 基础笔记 | 文本笔记、课程锚点、按课程归档 |
| 共创需求 | 提交需求、需求列表、投票、状态流转 |
| 订阅 | 连续包月、单月、年度；Web 支付先接微信/支付宝/Stripe 中至少一种 |
| 我的 | 账户、订阅状态、学习记录、笔记入口 |
| 后台 CMS | 课程管理、章节管理、上新日志、需求审核、用户/订单查看 |
| 埋点 | 访问、注册、试学、付费、学习、笔记、投票全链路 |

### 2.3 P1 不做

1. 不做原生 iOS / Android，只做响应式 Web。
2. 不做完整 AI 助教，只预留 AI 入口和数据结构。
3. 不做社区、评论区、直播、私信。
4. 不做创作者开放入驻，只做自制和签约内容。
5. 不做复杂游戏化，不做强制打卡。
6. 不做多人协作笔记。
7. 不做健康诊疗、处方建议、药品推荐。

### 2.4 P2 做

1. iOS / Android App。
2. 长辈模式完整体验。
3. 家庭订阅。
4. AI 笔记整理。
5. AI 课程内问答。
6. 笔记导出 PDF / Markdown。
7. 月度学习报告。
8. 结课卡分享裂变。

### 2.5 P3 做

1. 创作者生态。
2. 个性化学习路径。
3. 共学小组。
4. 离线缓存。
5. 学生认证价。
6. 礼品卡 / 企业团购。

## 3. 目标用户与优先级

### 3.1 用户分层

| 层级 | 年龄 | 核心需求 | P1 优先级 | 说明 |
|---|---:|---|---:|---|
| 学生/备考 | 18–24 | 雅思、四六级、考研、考公 | 高 | 目标明确，内容更新天然适合订阅 |
| 职场技能 | 22–40 | AI、办公效率、写作、编程、设计、剪辑 | 最高 | 付费决策快，是 P1 主力 |
| 中坚转型 | 35–45 | AI 提效、管理、行业转型、子女教育 | 中 | 年卡潜力高，但内容需要更可信 |
| 活力长辈 | 45–65 | AI 入门、手机使用、防诈骗、健康信息素养 | P1 试点，P2 放大 | 需要长辈模式和家庭代付支撑 |

### 3.2 P1 主人群

**P1 主人群：22–40 岁，愿意为 AI 实用技能和效率提升付费的人。**

原因：

1. 对“滚动更新”最敏感。
2. 对订阅制接受度更高。
3. 更容易通过短视频、小红书、SEO 转化。
4. 内容供给更容易快速更新。

### 3.3 P1 辅助人群

1. **18–24 备考用户**：用雅思/四六级做稳定刚需内容。
2. **45–65 长辈用户**：P1 只验证“家人代付 + 长辈 AI 入门课”的需求，不重投入。

## 4. 核心用户路径

### 4.1 游客转订阅路径

```text
访问首页
→ 理解“持续更新 + 全站订阅”
→ 看本周上新
→ 试学首章
→ 播放到关键处触发付费墙
→ 注册/登录
→ 订阅
→ 自动回到刚才学习位置
```

验收要求：

1. 付费墙不能打断用户首次理解，必须出现在试学有价值感之后。
2. 用户支付后必须回到原章节原进度。
3. 首页 3 屏内必须出现：价值主张、本周上新、订阅权益。

### 4.2 学习与笔记路径

```text
进入课程
→ 播放章节
→ 点击“记笔记”
→ 笔记自动带课程、章节、时间戳
→ 下次从笔记点击回到对应视频位置
→ 完课后生成学习记录
```

验收要求：

1. 笔记创建不应中断视频。
2. 时间戳必须可点击跳转。
3. 停订后笔记仍可访问。

### 4.3 共创需求路径

```text
订阅用户提交想学内容
→ 后台审核/合并重复需求
→ 进入需求广场
→ 用户投票
→ 需求进入评估中/已排期/制作中
→ 课程上线
→ 投票用户收到通知
```

验收要求：

1. 需求必须有状态，不允许长期“黑箱”。
2. 未采纳需求必须有原因。
3. 用户投过票的需求上线后必须能触达。

### 4.4 家庭代付路径（P2）

```text
子女购买家庭年卡
→ 邀请父母手机号
→ 父母收到短信/微信链接
→ 自动进入长辈模式
→ 一键开始“长辈 AI 入门”
→ 子女可看到总学习时长，但不能看具体笔记内容
```

验收要求：

1. 家庭成员隐私边界必须清晰。
2. 子女只能看到学习概览，不能看到课程笔记。
3. 长辈首次进入必须默认大字、大按钮、低密度。

## 6. 页面级 PRD 与验收标准

## 6.1 首页 / 发现页

### 目标

让用户在 30 秒内明白：

1. 这是一个订阅制学习平台。
2. 内容持续更新。
3. 用户可以投票决定新课程。
4. 学习和笔记是一体的。
5. 现在可以免费试学。

### 页面模块顺序

1. Hero 首屏。
2. 本周上新。
3. 热门课程。
4. 三条内容线：AI 技能 / 备考 / 生活。
5. 需求排行榜预览。
6. 学习 + 笔记演示。
7. 用户评价与数据证明。
8. 订阅方案。
9. FAQ。
10. 页脚。

### 关键组件

| 组件 | 要求 |
|---|---|
| Hero | 大标题、短副标题、主 CTA、次 CTA、动态渐变背景 |
| 上新卡 | 必须显示课程名、更新章节、更新时间、预计总时长 |
| 课程卡 | 标题、封面、标签、更新状态、学习人数、试学按钮 |
| 需求榜 | Top 5，显示票数、状态、投票按钮 |
| 订阅卡 | 连续包月默认高亮，但不得默认勾选额外服务 |

### 验收

1. 移动端首屏 CTA 可见。
2. 首屏 LCP 需要重点优化，Hero 动效低端设备自动降级。
3. 未登录用户点击试学可直接进入，点击付费再引导注册。
4. 订阅价格、续费规则、取消方式必须清晰展示。

## 6.2 课程库

### 筛选结构

1. 一级：全部 / AI 技能 / 备考 / 生活。
2. 二级：更新中 / 已完结 / 免费试学 / 入门 / 进阶 / 高阶。
3. 排序：推荐 / 最新更新 / 最多人学 / 最适合新手。

### 课程卡字段

| 字段 | 示例 |
|---|---|
| course_id | ai-office-001 |
| title | AI 办公效率入门 |
| subtitle | 从写邮件、做 PPT 到会议纪要 |
| category | AI 技能 |
| level | L1 入门 |
| status | 更新中 |
| update_text | 更新至第 8 讲 · 2 天前 |
| duration | 4 小时 20 分钟 |
| learners_count | 12,430 |
| free_lessons_count | 2 |

### 验收

1. 每门课必须显示更新状态。
2. 空结果页必须给出推荐课程。
3. 课程卡点击进入详情页时有共享元素动效，低端设备可关闭。

## 6.3 课程详情页

### 页面结构

1. 顶部封面。
2. 标题、简介、标签。
3. 讲师/内容责任人。
4. 关键数据：章节数、总时长、更新时间、学习人数。
5. 订阅/继续学习按钮。
6. 更新日志。
7. 课程大纲。
8. 适合谁 / 学完获得什么。
9. FAQ。
10. 相关推荐。

### 更新日志要求

每条更新日志包含：

```text
更新时间
更新类型：新增 / 修订 / 删除 / 纠错
影响章节
更新说明
内容责任人
```

### 验收

1. 更新日志必须放在大纲前或大纲旁边，强化“持续更新”。
2. 免费章节必须显式标记。
3. 付费用户显示“继续学习”，非付费用户显示“试学 / 订阅”。
4. 健康、财务、防诈骗课程必须显示审核人和免责声明。

## 6.4 播放器 / 学习页

### 功能

| 功能 | P1 | P2 |
|---|---:|---:|
| 视频播放 | 必做 | 优化 |
| 图文课件 | 必做 | 优化 |
| 进度记忆 | 必做 | 优化 |
| 倍速 | 必做 | 优化 |
| 章节切换 | 必做 | 优化 |
| 基础笔记 | 必做 | 优化 |
| 时间戳锚点 | 必做 | 优化 |
| AI 提问 | 预留入口 | 必做 |
| 离线缓存 | 不做 | 可做 |

### 桌面布局

```text
左侧：视频/课件主区域
右侧：笔记抽屉，默认宽 380px，可折叠
底部：章节进度、上一讲、下一讲
```

### 移动布局

```text
顶部：视频吸顶
中部：章节内容
底部 Tab：课件 / 笔记 / 目录
```

### 付费墙

触发位置：

1. 免费章节播放完。
2. 非免费章节开始前。
3. 图文课件滚动到免费部分底部。

文案规则：

1. 不制造焦虑。
2. 不倒计时逼单。
3. 明确解锁内容。
4. 明确取消方式。

示例：

```text
继续学习后面的 8 讲
包含：实战模板、案例拆解、每周更新
订阅后可解锁全站课程，笔记永久保留
```

### 验收

1. 切换章节不丢进度。
2. 创建笔记不暂停视频，除非用户主动暂停。
3. 断网时显示可恢复错误，不清空笔记草稿。
4. 非订阅用户不能通过接口直接获取付费视频 URL。

## 6.5 笔记

### P1 功能

1. 创建文本笔记。
2. 自动绑定课程、章节、时间戳。
3. 笔记列表按课程归档。
4. 搜索标题和正文。
5. 停订后仍可查看和删除自己的笔记。

### P2 功能

1. AI 整理成大纲。
2. 导出 Markdown / PDF。
3. 语音转文字。
4. 每日回顾卡。
5. 高亮课件并摘录到笔记。

### 笔记字段

| 字段 | 说明 |
|---|---|
| note_id | 笔记 ID |
| user_id | 用户 ID |
| course_id | 课程 ID |
| lesson_id | 章节 ID |
| timestamp_sec | 视频秒数，可为空 |
| title | 标题 |
| content_md | Markdown 内容 |
| source_text | 课件摘录，可为空 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 验收

1. 笔记必须自动保存。
2. 时间戳点击后跳转到视频对应秒数。
3. 用户停订后仍可打开笔记，但不能继续学习付费章节。

## 6.6 共创需求

### P1 功能

1. 用户提交需求。
2. 后台审核。
3. 重复需求合并。
4. 需求广场展示。
5. 订阅用户投票。
6. 需求状态流转。
7. 已上线需求跳转课程。

### 需求状态

```text
pending_review    待审核
collecting        征集中
evaluating        评估中
scheduled         已排期
producing         制作中
launched          已上线
rejected          未采纳
merged            已合并
```

### 投票规则

1. 仅订阅用户可投票。
2. 每个订阅用户每周 5 票。
3. 可对同一需求投多票，但最多 3 票。
4. 每周一 00:00 重置票额。
5. 异常投票进入风控队列。

### 排行榜分数

建议不要只按总票数排序。采用综合分：

```text
score = 总票数 * 0.45
      + 近7天新增票数 * 0.25
      + 付费用户投票数 * 0.20
      + 内容战略匹配分 * 0.10
      - 合规/制作风险惩罚
```

### 需求详情页

必须展示：

1. 需求标题。
2. 需求描述。
3. 当前票数。
4. 状态。
5. 状态日志。
6. 官方反馈。
7. 相似需求。
8. 如果已上线，显示对应课程。

### 验收

1. 投票后票数必须即时变化。
2. 用户能看到自己投过的需求。
3. 需求被合并后，原投票不能丢失。
4. 未采纳必须填写原因。

## 7.3 Entitlement 状态机

```text
free
 → trial
 → active
 → grace_period
 → billing_retry
 → active / expired

active
 → canceled_but_active_until_period_end
 → expired

active
 → refunded / revoked
```

### 后端原则

1. 客户端不判断权益，只展示服务端返回的权益快照。
2. 所有支付回调必须幂等处理。
3. 订单、订阅、权益分表，不要混在一个表里。
4. Web / iOS / Android 统一映射到同一套 entitlement。
5. 必须支持恢复购买。
6. 必须记录 webhook 原始 payload，便于排查。

### 渠道注意

1. **iOS App 内数字内容解锁原则上走 IAP**；Web 已购订阅可以在 App 内登录后使用，但 App 内购买入口要遵守 App Store 规则。
2. **中国区 App Store 费率需按 2026 年 3 月后的新政策重新核算**，不要继续在财务模型里固定写 30%。
3. **Google Play 订阅要按 base plan / offer / entitlement 生命周期设计**，不要只用一个“paid=true”。
4. 国内 Android 商店包和官网包的支付策略要分开，避免商店审核风险。

## 12. AI 功能规划

## 12.1 P1 原则

P1 不做复杂 AI 功能，只做三件事：

1. 数据结构预留。
2. UI 入口预留。
3. 内容转写稿和课件结构化，为 P2 做准备。

## 12.2 P2 AI 功能

| 功能 | 说明 | 风险控制 |
|---|---|---|
| AI 笔记整理 | 将用户速记整理成大纲 | 不篡改原意，保留原文 |
| 课程内问答 | 基于当前课程转写稿回答 | 必须引用课程段落 |
| 周复习摘要 | 汇总本周笔记 | 用户确认后保存 |
| 学习路径推荐 | 根据目标和进度推荐下一课 | 不承诺结果 |
| 长辈语音助手 | 语音搜索课程/记笔记 | 避免诱导支付 |

## 12.3 AI 健康类限制

禁止：

1. 诊断疾病。
2. 推荐处方药。
3. 给出用药剂量。
4. 替代医生判断。
5. 根据症状直接判断严重程度。

允许：

1. 帮用户整理就医前信息。
2. 提醒记录症状发生时间、持续时长、既往病史。
3. 解释“如何和医生沟通”。
4. 提醒紧急情况及时就医。
5. 给出健康信息识别方法。

统一免责声明：

```text
本内容仅用于健康信息素养学习，不构成诊断、治疗或用药建议。身体不适请及时咨询正规医疗机构。
```

| 抽屉展开 | 240–320ms | reduced-motion 改淡入 |
| 页面转场 | 320ms | reduced-motion 关闭 |
| Hero 渐变 | 20s 慢速 | 低端设备静态 |
| 数字滚动 | 600ms | 直接显示最终数字 |

## 13.6 长辈模式验收

1. 首次进入可选择长辈模式。
2. 设置页必须有“切换长辈模式”。
3. 长辈模式不出现诱导下载、诱导付款、倒计时弹窗。
4. 字号可调。
5. 行距至少 1.3 倍，建议正文 1.6 倍以上。
6. 按钮点击区 ≥56pt。
7. 语音输入入口显著。
8. 付款页必须简洁，不能默认勾选附加项。
9. 客服入口显著。
10. 家人协助不能泄露笔记内容。

## 15. 风险与处理

| 风险 | 处理 |
|---|---|
| 方向过宽 | P1 只验证 AI 技能 + 备考，生活线只做试点 |
| 内容供给跟不上 | 内容排期后台 + 固定上新日 + 延期说明 |
| 用户觉得像普通课站 | 强化更新日志、需求榜、投票上线通知 |
| 长辈用户不会用 | P2 做长辈模式、语音输入、家庭代付、客服入口 |
| 健康内容踩线 | 全部改为健康信息素养，不做诊断和用药建议 |
| 订阅审核风险 | iOS / Android 分渠道遵守商店规则，Web 购买和 App 内权益打通 |
| 笔记数据丢失 | 自动保存、版本记录、导出、备份 |
| 需求榜被刷 | 订阅用户投票、周票额、设备/IP 风控、后台审核 |
| 盗版录屏 | HLS 加密、短时 URL、动态水印、版权投诉流程 |
| 高级感变成空 | 设计系统 + 组件验收 + 真机性能测试 |

## 21. 最终建议

这不是一个“再做一个知识付费平台”的项目。真正的卖点必须压在四个地方：

1. **持续更新**：每门课都要让用户看到“这不是死课”。
2. **用户共创**：用户投票真的影响课程排期。
3. **笔记资产**：用户沉淀越多，留存越强。
4. **全龄可信**：年轻人觉得高级，长辈觉得简单，子女觉得放心。

P1 先别贪大。只要 Web 版把“试学 → 订阅 → 学习 → 记笔记 → 投票 → 上新提醒”闭环跑通，这个产品就有继续扩大的基础。
